# UAT panel — `chatlab` v1.0.0-rc.1

- **Date:** 2026-04-30
- **Project:** `chatlab` v1.0.0-rc.1
- **Panel:** six downstream-role evaluators (data scientist, data annotator, tester, ML engineer, data analyst, academic NLP researcher).
- **Scope:** the published surface as it stands on `main` after the TRB closeout review of the same date — at-rest encryption, retention sweep, structured logger, privacy banner, design system, markdown rendering, `custom` provider all live; capability `0007-eval-harness` is drafted for v1.1, not implemented.
- **Methodology:** each evaluator was given the User Guide + recipes + a working install (Ollama + one cloud provider) and asked to spend ~30 minutes attempting their **real** workflow. Notes captured below are condensed; the user-stories section at the bottom is the durable artifact.

## Reviewer note

This is a **UAT panel** report, not a TRB review. The TRB looks at engineering health from the inside; UAT looks at adoption friction from the outside. Both are filed under `docs/reviews/` because they share the same "snapshot at a release boundary, read-only after publication" lifecycle.

## Executive summary

| Item | Result |
| --- | --- |
| **Adoption friction** | **Low** for ML engineers and testers; **medium** for data scientists and data annotators; **high** for data analysts and NLP researchers. |
| **Wedge holds** | Yes for the "build an agent locally and capture feedback" loop. The `custom` provider was the strongest signal across all six roles. |
| **Top regret** | No batch / scenario-replay surface. Every role hit it from a different angle (replay for testers, golden-set for ML, time-window aggregations for analysts). The eval-harness draft (capability `0007`) covers part of this; the rest is new scope. |

---

## Cross-cutting strengths

The six roles converged on four things.

1. **The `custom` provider is the unlock.** Every evaluator independently said some version of "ah, this is for testing my agent, not consuming an LLM" within the first five minutes. The wedge is legible.
2. **JSONL export is honest.** No hidden fields, no transformation step, `schema_version: 1` documented and stable. The data scientists, ML engineers, and analysts all moved straight to `jq` / pandas without asking.
3. **At-rest encryption + privacy banner felt grown-up.** The annotators and the NLP researcher specifically called out the banner as "the kind of thing I don't expect in a v1.0 OSS tool". The encryption story landed credibly even without a formal third-party audit.
4. **Markdown rendering everywhere** (chat bubbles + chat notes preview) made the agent's outputs **readable**. Code blocks survive copy-paste; tables don't get mangled. This was a quiet feature for the engineers but a loud one for the annotators ("I can finally read the code my agent emits").

## Cross-cutting weaknesses

Three things came up in **four or more** of the six roles.

1. **No batch / scenario / replay surface.** The product is one-message-at-a-time. Replaying a curated 50-prompt set against a new agent is manual: open chat, paste, scroll, repeat. Eval harness (capability `0007`) addresses this for the "compare against baseline" angle but doesn't cover replay-for-debugging.
2. **No analytical surface beyond raw export.** No `GET /v1/feedback/summary?group=theme,agent_version`, no time-bucket counts, no per-workspace dashboard. Analysts and researchers rebuilt these on top of DuckDB and resented having to.
3. **Single-rater bias is structural.** The corpus assumes one ground-truth rater per row. Annotators and academic researchers both flagged it as a publishability blocker (no inter-annotator agreement, no per-row rater identity, no annotation queue for second-pass review). ADR 0007's evolution log forecasts this as schema-additive (one row per `(message_id, rater)`), but the UI affordance is missing.

## Per-role notes (condensed)

### 1. Cientista de Dados

**Background:** explores corpora to find failure-mode clusters and feature-engineer prompts.

**Strengths felt:** DuckDB workspace + `read_json_auto` on the export = "I can stay in DuckDB, I don't need to go through pandas first"; `agent_version` slicing.

**Weaknesses felt:**

- No notebook-friendly Python helper (`from chatlab import Workspace` → DataFrame). Has to curl + parse.
- No embedding helper. Wants `/v1/messages/{id}/embedding` or at least a documented recipe.
- No anonymization/redaction pass on export — has to write the regex herself.
- Per-workspace stats (counts, ratings dist, agent diversity) require a query — wishes for `chatlab stats` CLI.

### 2. Anotador de Dados

**Background:** rates assistant replies for several hours per day; cares about throughput per session.

**Strengths felt:** 👍/👎 with one click; the markdown preview in chat notes; auto-save on blur; `***last4` masking on API keys means the token never appears in copy-paste session logs.

**Weaknesses felt:**

- **No keyboard shortcuts.** `j`/`k` to move through assistant bubbles, `u`/`d` for thumbs, `n` for note. Today everything is mouse + click.
- **No "show me the next unrated" filter.** Has to scroll.
- 280-char comment limit feels arbitrary for nuance ("the agent confused order ID with tracking ID and offered a refund instead of dispatch info" is already 100 chars before specifics).
- No "flag for review" → reviewer-queue handoff; the `flagged_for_review` field exists in export but no UI to set it.
- No batch operations (rate-up all replies in this chat, mark all as reviewed).

### 3. Testador / QA

**Background:** drives the agent through fixed scenarios to reproduce bugs the engineering team is debugging.

**Strengths felt:** workspace segregation lets him keep "scenario A" and "scenario B" independent; `memory` adapter means a fresh test starts in <1 second; `CHATLAB_HOME` override is portable across machines.

**Weaknesses felt:**

- **No scenario replay.** "Send these 12 messages in this order to the agent" is a manual ritual.
- **No assertion DSL.** Wants to say "the assistant reply should contain 'order #' or `status: refund`"; today he reads each reply by eye.
- **No shareable workspace bundle.** "Send this scenario to a colleague" = tar `~/.chatlab/data/<uuid>.db` and email — and even then the master key for at-rest encryption doesn't travel, so the agent's API key is gone.
- No comparison view (run same prompt against agent A and B side-by-side). Has to flip between chats.

### 4. Engenheiro de Machine Learning

**Background:** owns the fine-tuning loop; consumes the JSONL export to train DPO / SFT models.

**Strengths felt:** `schema_version: 1` contract; `agent_version` per row; the `failure_category` + `flagged_for_review` optional fields; ADR 0007 evolution log gives him confidence the schema won't churn.

**Weaknesses felt:**

- **No token / cost in the export.** Has to estimate from message length, which is wrong for tokenizers like tiktoken.
- **No deterministic mode per agent.** Temperature is hardcoded `0.7` in the runner; for golden-set runs he wants `0` to make diffs reproducible.
- **No streaming support** — for long replies he can't capture partial outputs as the user would see them.
- **No `temperature` / `top_p` / `seed` knobs per agent profile.** One temperature for everything is the wrong default.
- The eval harness (`0007`) is drafted but he's been waiting on it for the side-by-side compare he actually wants for v1.0.

### 5. Analista de Dados

**Background:** builds dashboards on chat-quality KPIs for a product team.

**Strengths felt:** filtered exports (`?rating=down&since=...`) reduce the volume he has to ingest; DuckDB analytic adapter is "exactly the right primitive"; the JSONL streaming means he can pipe directly into BigQuery's `bq load`.

**Weaknesses felt:**

- **No aggregation endpoint.** `GET /v1/feedback/export` then `jq` then re-aggregate is fine for dev; for a Grafana dashboard he wants `GET /v1/feedback/summary?group=theme,agent_version,rating&bucket=1d` with a structured response.
- **No CSV export.** He has Excel users on his team; JSONL is friction for them.
- **No webhook on rating events.** He poll-imports nightly because there's no push.
- **No schema_version bump signal in the export header.** A response header `X-Chatlab-Schema-Version: 1` would let him hard-fail his pipeline on drift.

### 6. Pesquisador Acadêmico de NLP

**Background:** builds annotated corpora to publish benchmarks; needs reproducibility and ethics-committee defensibility.

**Strengths felt:** MIT license; `schema_version` lock; `agent_version` capture; `theme` field as a built-in stratum; the privacy banner as evidence of consent surface.

**Weaknesses felt:**

- **No system_prompt in the export.** He wants the full ground truth: `system + user + assistant + rating`. Today the system prompt is implicit in `agent_version`, but he can't reconstruct it after the fact.
- **No DOI / citation helper.** Submitting a corpus alongside a paper requires a stable identifier and a BibTeX block; he writes both by hand.
- **No multi-rater workflow** — inter-annotator agreement (Cohen's κ, Krippendorff's α) is a publishability gate; without it the corpus is one-rater anecdote.
- **No consent audit log.** The privacy banner is shown once per session and dismissed; for IRB review he needs a record of *who* dismissed *when*.
- **No subset-by-ethics export.** "All rows the user explicitly opted in to share" is a useful filter.
- The default `temperature: 0.7` plus no seed means even the same prompt + agent + theme produces different outputs — bad for reproducibility studies.

## User stories — backlog

Each story is in the format **As [persona], I want [feature] so that [outcome]**, with a rough effort estimate (`S` / `M` / `L`) and a proposed target. Effort is from the maintainer's seat: `S` = 1–2 days, `M` = 1 week, `L` = 2+ weeks.

### v1.1 candidates (high impact, well-scoped)

1. **As an ML engineer, I want token + cost approximation in the JSONL export** (`prompt_tokens`, `completion_tokens`, `cost_estimate_usd` on `agent_message`) so that I can attribute training-set value to spend. **Effort: M.** Already in v1.1 ROADMAP; bumps `schema_version` to 2.

2. **As an ML engineer, I want per-agent `temperature` / `top_p` / `seed` knobs** so that golden-set runs are reproducible. **Effort: S.** Touches `Agent` schema, `runner.ts`, AgentForm UI.

3. **As a data annotator, I want keyboard shortcuts** (`j`/`k` to navigate assistant bubbles, `u`/`d` for thumbs, `n` to focus the note textarea) so that I don't switch hands. **Effort: S.**

4. **As a data annotator, I want a "show me unrated assistant replies" filter** in the chat list so that I work through the queue without scrolling. **Effort: S.** Add `?rating=missing` filter to `GET /v1/chats` listing or a separate `/v1/messages?has_feedback=false` endpoint.

5. **As a tester, I want to export a workspace as a portable bundle** (a `.tar.gz` with the workspace `.db` + a stripped manifest of agents without API keys) so that I can ship a reproducible scenario. **Effort: M.** Pairs with item 6.

6. **As a tester, I want to import a workspace bundle** so that scenarios travel between machines. **Effort: M.** This is also Open Question 2 of capability `0001-workspaces` (workspace duplicate, listed in v1.1).

7. **As a data scientist, I want a `chatlab stats` CLI** that prints per-workspace summary (chats, messages, ratings distribution, agent diversity, theme histogram) so that I have a one-glance dashboard without writing SQL. **Effort: S.**

8. **As a data analyst, I want `GET /v1/feedback/summary?group=theme,agent_version,rating&bucket=1d`** returning structured aggregations so that I don't ingest the full corpus into Grafana. **Effort: M.** Backend SQL on each adapter; thin HTTP shape.

9. **As a data analyst, I want CSV export** as an alternative content-negotiation on `/v1/feedback/export?format=csv` so that non-technical users can open it in Excel. **Effort: S.** Pure transformation of existing JSONL.

10. **As an NLP researcher, I want the export to include `system_prompt`** as a top-level field on `prompt_message` so that the corpus is self-contained. **Effort: S.** Bumps `schema_version` to 2 (additive but worth grouping with item 1).

### v1.2 candidates (require multi-component design)

11. **As an ML engineer, I want a side-by-side "compare two agents" view in the UI** — same prompt, same theme, two replies rendered next to each other, with an A/B preference rating. **Effort: L.** Touches Core (parallel inflight), UI (split view), capability spec (feedback row gains `pair_id` + `chosen_id`).

12. **As an annotator, I want a multi-rater workflow** — when a chat has been rated by user X, optionally show it to user Y for cross-rating; export captures one row per `(message_id, rater)`; UI surfaces inter-annotator agreement metrics. **Effort: L.** Schema bump (`schema_version: 2` or `3`); auth model gains user identity (today auth is a single Bearer token, no user concept).

13. **As an NLP researcher, I want a `chatlab cite --workspace <id>` command** that emits a BibTeX block + a deterministic content hash of the corpus so that the publication is citeable and reproducible. **Effort: S.** Once user-identity (item 12) lands, this is a one-evening feature.

14. **As an NLP researcher, I want a consent audit log** — each privacy-banner dismissal logged with timestamp + (when item 12 lands) user identity. **Effort: S.** Add `consent_acks` namespace to the storage adapter.

15. **As a tester, I want a scenario replay** — record a chat session (sequence of user messages) and replay it against a different agent, capturing all assistant replies for diff. **Effort: M.** Capability spec needed; effectively a thin subset of capability `0007-eval-harness` extended to multi-turn.

16. **As a tester, I want an assertion DSL** — `assertions:` block in the scenario YAML with patterns like `assistant.content matches /order #\d+/`. **Effort: M.** Pairs with item 15.

17. **As an analyst, I want a webhook on `chat.assistant-replied` and `feedback.set`** so that downstream pipelines push instead of polling. **Effort: S.** The WS gateway already broadcasts these; webhook is a thin re-emit.

### v2 / out-of-scope candidates (revisit when conditions change)

18. **As a data scientist, I want `/v1/messages/{id}/embedding`** to materialize sentence-level embeddings for clustering. **Effort: L.** Picks an embedding model — out of scope while the project is local-first, deserves an ADR before implementation.

19. **As an annotator, I want comments to be 2 KB instead of 280 chars.** **Effort: S** — but per ADR 0007 the 280 cap is deliberate ("forcing a tweet-sized rationale keeps annotators honest"). Decision needs an ADR superseding 0007 if we change it.

20. **As an analyst, I want a built-in dashboard inside the Web UI** (small charts on the home tab — ratings/day, agent_version distribution). **Effort: L.** Risks scope-creep; for v2.

21. **As an NLP researcher, I want federated multi-installation eval** so that two labs can run the same golden set and compare. **Effort: L+.** Implies hosted instance, which ADR 0011 deferred. Revisit only if ADR 0011's reopening criteria fire.

## Recommendation to maintainer

The panel converged on three pieces of immediate, low-risk follow-up that punch above their weight:

- **Items 1, 2, 10** together produce a single `schema_version: 2` bump with token / cost / system_prompt / temperature-aware export. Three weeks of work in one cohesive release.
- **Items 3, 4, 7** are all small UI / CLI affordances that compound: an annotator with `j`/`k` + filter, plus a maintainer with `chatlab stats`, suddenly looks two-versions newer.
- **Items 11 + 12** are the v1.2 anchor — multi-rater + side-by-side compare turn chatlab from "developer's workbench" into "team's workbench" and unlock the academic-research persona properly.

The eval harness draft (`0007`) covers items 15+16 partially. Reviewers recommend completing it as the first v1.1 capability before grouping the others.

## Readiness statement

> The panel finds `chatlab v1.0.0-rc.1` **ready to use** for the developer-as-evaluator workflow it was designed for. For team-shared annotation, analytics, and reproducible academic study, the panel finds it **promising but not yet sufficient** — items 11, 12, 14 are the gates. The recommendation is to ship `v1.0.0` with the GA blockers from the TRB review closed, and immediately open a `v1.1` planning cycle that prioritizes items 1–10 of this report.
