# TRB review — `chatlab` v1.0.0-rc.1

- **Date:** 2026-04-30
- **Project:** `chatlab` v1.0.0-rc.1 (local development platform for chat agents)
- **Panel:** 14 senior reviewers across product, engineering, security, design, ops, data, ML, legal, and adjacent disciplines.
- **Scope:** all artifacts on `main` at review time — `src/`, `test/`, `docs/`, `docs-site/`, `.github/`, configs, and the published Docusaurus site.

## Reviewer note on framing

The convening prompt described the project as "Emulador de WhatsApp para desenvolvimento local de Chat Agents". The artifact under review is **not** that — `chatlab` discarded WhatsApp Cloud-API parity and rebrands as a generic local platform for chat agents. The fact that the framing in the review prompt itself used the pre-pivot vocabulary is a finding the panel returned to in the verdict (see [Critical challenge](#critical-challenge)).

## Executive summary

| Item | Result |
| --- | --- |
| **Maturity score** | **7.0 / 10** |
| **Critical challenge** | Lack of a clear differentiation narrative, backed by operational guarantees. |
| **Readiness** | Proceed to v1.0 GA **conditional** on five blockers. |

### Five GA blockers (from the verdict)

These five recommendations were ruled mandatory before promoting `v1.0.0-rc.1` → `v1.0.0` and announcing publicly:

1. **Encrypt provider API keys at rest.** Today they're plaintext in `~/.chatlab/data/<uuid>.{db,duckdb}`. Source: Security Engineer.
2. **Implement automatic feedback retention sweep.** `CHATLAB_FEEDBACK_RETENTION_DAYS` is documented as enforced; `feedback.sweepOlderThan` is currently manual. Source: Data Engineer.
3. **Adopt a structured logger** (pino or similar) gated by `CHATLAB_LOG_LEVEL`, with `chat_id` / `workspace_id` enrichment on every relevant log line. Source: SRE.
4. **Write a "Why chatlab" section** in `README.md` and the Docusaurus home, comparing chatlab against three named alternatives. Source: Product Manager + Technical Writer.
5. **Privacy banner on first run** (CLI banner + UI banner) when the active workspace contains a cloud-provider agent. Source: Legal Advisor + ML Engineer.

Other recommendations (E2E suite, axe audit, eval harness, compose example, perf benchmarks) are tracked for v1.1 and **do not block the GA**. The full list is in the [action register](#action-register).

## Individual pareceres (condensed)

### 1. Product Manager

**Strengths:** post-pivot positioning is coherent; MVP scope (capabilities 0001–0006) shipped as a unit; personas Bruno/Camila/Diego are distinct and map to specific capabilities; ROADMAP separates v1.0 / v1.1 / v1.2+ with objective scope per release.

**Risk (Alta):** the market is crowded by LangChain/LangSmith, PromptFlow, Helicone, Langfuse, OpenAI Playground. The wedge ("local-first with JSONL export for fine-tuning") is defensible but the docs don't argue it — the homepage describes mechanics, not differentiation.

**Recommendation:** add a "Why chatlab and not <X>?" section to `README.md` and the Docusaurus home, naming three competitors and the swap-trigger for each. Acceptance: a non-technical reader, after 60 seconds, can state in one sentence when chatlab is the right pick.

### 2. Software Engineer

**Strengths:** stack alignment (Node 22 + TS + Express + React + Vite); 13 ADRs anchor decisions; the pluggable `StorageAdapter` with shared `_battery.ts` is a rare example of multi-implementation polymorphism done right; `Core` as event emitter centralizes the WS broadcast.

**Risk (Alta):** `Core` is a process-global singleton with mutable state and `await core.getActiveAdapter()` — the "swap during request in-flight" semantics are not tested. AgentRunner has a 60 s timeout but no retry, no idempotency key, no dead-letter — provider 5xx silently drops the assistant turn.

**Recommendation:** add a Vitest spec that fires `chat.user-message-appended`, swaps the active workspace before the reply lands, and asserts the original reply still persists in the original workspace (or fails with a captured error). Metric: zero silent `agent.failed` under 100 sequential swaps.

### 3. Security Engineer

**Strengths:** bind-safety check (exit 78 without token on non-loopback host) is exemplary — explicit, documented, tested. CodeQL + Gitleaks + OSV-Scanner gate every PR. Dependabot daily. SECURITY.md has private-disclosure flow with 7/14-day SLAs.

**Risk (Crítica):** provider API keys are stored as **plaintext** in `~/.chatlab/data/<uuid>.{db,duckdb}`. Any process with the same UID can read them. The default permissive Bearer auth (any non-empty string) doesn't stop a malicious local process from listing workspaces and exfiltrating chats via `/v1/feedback/export`. No rate-limiting. Media upload doesn't validate MIME against magic bytes nor cap size in any documented place. For commercial / LGPD-scoped use this is non-compliant.

**Recommendation:** encrypt `api_key` at rest with a key derived from `CHATLAB_HOME` + a master key kept in the OS keychain (macOS Keychain, Linux Secret Service, Windows Credential Manager); CI fallback documented. Acceptance: `sqlite3 ~/.chatlab/data/X.db "SELECT api_key FROM agents"` returns ciphertext, not `sk-...`.

### 4. UI/UX Designer

**Strengths:** design system adoption (ADR 0013) brought OKLCH tokens, light/dark + 3 density modes, IBM Plex typography, reusable primitives (`.btn`, `.bb`, `.card`). Theme toggle persisted in `localStorage`. Inline bootstrap script in `index.html` avoids flash. Header IA (logo + workspace switcher + tabs + theme toggle) is plausible.

**Risk (Média; Alta if WCAG ambition is real):** WCAG audit never executed. OKLCH contrast values not verified against AA (4.5:1 text, 3:1 UI). `axe-core/playwright` is deferred (ADR 0010 §5). Some `<DevDrawer>` icon-only buttons miss `aria-label`. Mobile layout breaks below 480 px viewport (`--col-list: 300px` is non-responsive).

**Recommendation:** run a manual axe DevTools pass over the 5 main screens (Chats, Admin/Workspaces, Admin/Agents, AgentForm, empty state) in light + dark; open issues for findings ≥ moderate. Metric: zero `serious` / `critical` violations before v1.0 GA.

### 5. DevOps Engineer

**Strengths:** the new `ci.yml` closes the PR gate (typecheck + test + coverage + build + redocly). `lint-docs.yml` exercises OpenAPI + Mermaid + lychee links + Docusaurus build. `release.yml` does npm + multi-arch Docker + GitHub Release. `docs-deploy.yml` publishes to Pages via `actions/deploy-pages`. Dependabot active. Concurrency cancels in-progress.

**Risk (Média):** bus factor of one — `DOCKERHUB_USERNAME` / `DOCKERHUB_TOKEN` / `NPM_TOKEN` belong to a single maintainer. No staging environment. `release.yml` and `lint-docs.yml` use `npx -y @redocly/cli@latest` without a version pin — a CLI breaking-change ships broken releases. Pages has no canary.

**Recommendation:** pin `@redocly/cli` to a stable major (`@1.27.0` or current LTS) in `release.yml` and `lint-docs.yml`; let Dependabot bump it. Acceptance: `git grep "@redocly/cli@latest"` returns zero hits.

### 6. SRE

**Strengths:** health probes (`/healthz`, `/readyz`) are well-defined and documented. Bind-safety prevents accidental network exposure. WS reconnect uses exponential backoff (0.5 s → 30 s cap). AgentRunner has 60 s timeout + inflight counter.

**Risk (Média for local; Alta for any team-shared deployment):** zero structured observability. Logs are `process.stderr.write` of free-form text — no JSON, no correlation-id per request, no `/metrics` endpoint, no OTel tracing. No SLO/SLI defined. Graceful shutdown does not document inflight-agent-call drain.

**Recommendation:** adopt pino as the logger (matches the Node + ESM + zero-config preference of the rest of the stack), level controlled by the existing `CHATLAB_LOG_LEVEL`, every log line carries `chat_id` / `workspace_id` when relevant. Metric: 100 % of logs are JSON-parseable; every error log carries a `chat_id` when the error is chat-scoped.

### 7. Data Engineer

**Strengths:** ADR 0007 locks the export schema — numbered `schema_version`, evolution log, JSONL streaming, additive-only for non-breaking changes. Server-side query filters (`rating`, `since`, `chat_id`). Retention configurable via `CHATLAB_FEEDBACK_RETENTION_DAYS`. DuckDB adapter enables `read_json_auto` analytics directly on the export.

**Risk (Alta):** retention sweep is **manual** in v1.0. Documentation promises "automatic deletion at 90 days" but `feedback.sweepOlderThan` is only invoked if the operator scripts it. This is a documented LGPD promise the code doesn't keep. Separately, `agent_version` is computed at write time as `<provider>:<model>` — if the operator renames the agent or PATCHes `model`, prior rows can't be reattributed.

**Recommendation:** implement automatic sweep in v1.0 GA as a daily timer in `Core`, idempotent, observable via structured log (depends on rec 6). Acceptance: starting chatlab with `CHATLAB_FEEDBACK_RETENTION_DAYS=1` deletes feedback rows older than 24 h without manual intervention.

### 8. ML Engineer / AI Specialist

**Strengths:** six providers via two HTTP shapes (`openai-compat` + `anthropic`) is the right scope/coverage trade-off. `agent_version` per export row enables slicing. `exporting-feedback.md` walkthrough (jq, pandas, HuggingFace `datasets`, DuckDB) is useful for downstream consumers.

**Risk (Alta):** no eval harness, no drift detection, no guardrails. A prompt change has no objective measurement loop — the only feedback is "the developer reads replies by hand". No mitigation for prompt injection or exfil via tool-call (tool-calling is deferred but `system_prompt` already accepts arbitrary input). No consent flow informing the user that "this conversation will be exported as training data" — relevant when the rater is the Camila persona, not the developer.

**Recommendation:** specify capability `0007-eval-harness` (regression suite of 10 prompts, baseline outputs, diff per agent/version) for v1.1 — must land before tool-calling. Metric: dev runs `npm run eval -- --agent <id>` and gets a textual diff table against baseline.

### 9. Business Analyst

**Strengths:** capability specs follow a consistent template (Status / Authors / Related ADRs / Depends on / Summary / Motivation / User stories / Behavior / Verification). Open Questions tracked explicitly per spec. ROADMAP scoped per release with numbered items. `Status` field formalizes lifecycle (`Draft` / `Accepted` / `Implemented` / `Superseded`).

**Risk (Média):** loose Definition of Done. Flipping `Status: Draft → Implemented` is a Markdown edit with no enforced checklist. Each spec's `Verification:` section lists scenarios as bullets, but those bullets are **not** linked to specific Vitest test IDs — that linkage was deliberately dropped in the ADR 0010 rewrite. Reviewers cannot verify that a "Implemented" capability still satisfies its own promises.

**Recommendation:** add an "Acceptance" section to the capability spec template with three required entries: Vitest test ID(s), OpenAPI operation(s), User Guide section. Apply retroactively to all 6 active specs. Acceptance: every spec marked `Implemented` points at ≥ 1 test that would fail if the feature were reverted.

### 10. Technical Writer

**Strengths:** the documentation is among the strongest seen for an OSS project of this size. Concise README, quickstart, exhaustive `recipes.md`, narrative user guide, capability specs, ADRs, Docusaurus site, OpenAPI rendered via Redoc, design system in its own folder. Recently added cookbook / providers / troubleshooting / exporting-feedback closed real gaps.

**Risk (Baixa):** Portuguese is absent — personas have BR names, prompts in examples are PT, Maritaca is a first-class provider, but every page is English-only. Architectural sequence diagrams are absent (Mermaid is used elsewhere; the user-message → AgentRunner → provider → assistant-replied flow is described in prose).

**Recommendation:** add a sequence diagram (Mermaid) to `docs/ARCHITECTURE.md` covering the message-to-reply flow. Acceptance: a reader can map every arrow in the diagram to a file in `src/` within 5 minutes.

### 11. QA Engineer

**Strengths:** 82 Vitest specs, 80/80/80/65 coverage gate, full suite under 2 s. Storage battery shared across three adapters. Playwright wired for `docs:capture`. ADR 0010 is honest about scope (no E2E, no axe, no contract suite — by choice).

**Risk (Alta for the UI; Média for src/):** `src/ui/**` is excluded from coverage; there is no E2E regression suite; there is no integration test against a real Ollama. The UI is exercised only by humans and by screenshot capture (which validates pixels, not behaviour). Branch coverage at 65 % is honest about defensive paths but conceals genuine gaps.

**Recommendation:** add one Playwright spec under `test/e2e/` (new folder) that creates a workspace, configures an Ollama agent, opens a chat, sends a message, awaits the reply, rates it, exports feedback, asserts JSONL shape. Default-skipped via `CHATLAB_TEST_E2E=1` env gate. Metric: smoke E2E completes < 30 s against a local Ollama when the gate is open.

### 12. Performance Engineer

**Strengths:** Vitest sub-2 s. Vite UI build sub-700 ms. Bundle 21 KB CSS / 56 KB JS gzip. `better-sqlite3` synchronous calls are faster than async pools at dev volumes. WS broadcast tolerates dead clients without blocking.

**Risk (Média):** zero documented benchmarks. No answer to "1 k chats? 10 k messages? 100 simultaneous ratings?". DuckDB is sold as analytical but per-row write throughput can be 10×–100× lower than SQLite — no comparison published. AgentRunner serializes per chat (correct) but two chats in the same workspace can fire two simultaneous calls to the same upstream provider — without internal rate-limit, easy to trip 429s.

**Recommendation:** add `test/perf/storage-bench.test.ts` (skipped by default, env-gated) measuring insert + read of 10 k messages per adapter; print a table; commit the latest numbers to `docs/ARCHITECTURE.md` "Performance characteristics" section. Metric: numbers are republished on every release.

### 13. Cloud Architect

**Strengths:** local-first is a deliberate choice (ADR 0011 — Hosted instance Deferred). Docker multi-arch (linux/amd64 + linux/arm64) via QEMU + Buildx. Image base `node:22-bookworm-slim`, non-root `chatlab` user, embedded healthcheck. Reopening criteria for hosted instance are explicit and falsifiable.

**Risk (Baixa given local-first; Alta if team-share emerges as a v1.x ask):** no "self-host on $cloud" recipe. For Camila-on-a-small-team-on-a-droplet, there is no Docker Compose with TLS, no Helm chart, no terraform/pulumi. The "expose with `CHATLAB_REQUIRE_TOKEN`" path is manual and easy to misconfigure (weak token, no TLS).

**Recommendation:** add `docs/distribution/compose.example.yml` (already mentioned as planned in `docker.md`) with chatlab + Caddy + token generated by the entrypoint. Acceptance: `docker compose up` on a fresh VM exposes chatlab over HTTPS with Let's Encrypt automatically.

### 14. Legal Advisor

**Strengths:** MIT license; `SECURITY.md` private-disclosure path with SLAs; `docs/legal/data-handling.md` includes a DPA template, LGPD/GDPR posture, recommendations for commercial adopters (synthetic data, retention, redaction, RoPA, sub-processors). Trademark posture clean post-pivot.

**Risk (Alta for any deployment with real PII):** six potential sub-processors (OpenAI, Anthropic, DeepSeek, Gemini, Maritaca, Ollama-remote) and the product enforces nothing — no consent surface, no data-flow log, no redaction. When Camila types "cliente João, CPF 123, reclamou…", the text goes plain to the configured provider — chatlab is the controller under the LGPD "decides means and ends" test. The data-handling doc documents this but documentation does not equal consent.

**Recommendation:** add a persistent UI banner ("Conversations are sent in cleartext to the configured provider. Do not use real personal data without redaction.") plus a CLI startup warning when the active workspace contains ≥ 1 cloud-provider agent. Acceptance: a user cannot create their first cloud-provider agent without seeing the warning once.

## Critical challenge

The panel converged on three findings that are facets of one problem:

1. **Weak external narrative** (Product Manager + Technical Writer): the docs describe mechanics, not differentiation versus mature competitors.
2. **Underdeveloped operational guarantees** (Security + SRE + Legal + Data): plaintext API keys, manual retention sweep, free-form logs, no consent surface — any "use this commercially" narrative collides with the code.
3. **The TRB convening prompt itself used the pre-pivot vocabulary** ("Emulador de WhatsApp"). If the convening party reaches for the old name, the cold reader will too. The story has to be told deliberately, not assumed.

These are one disease — **the project knows what it is, but does not yet communicate or guarantee that to the world**.

## Readiness statement
